# Itinerary Planning

## Overview

该 Skill 用于把“去哪里玩”的想法变成按天可执行的完整行程，强调节奏合理、路线上顺、预算可控、体验稳定。
输出默认包含“标准版日程 + 轻松版备选”，便于用户按体力与天气灵活切换。

## 适用触发场景

- “我去某地 3 天，帮我安排怎么玩”
- “带老人/孩子出游，行程不要太赶”
- “预算有限，想覆盖核心景点”
- “想美食 + 文化 + 夜景都兼顾”
- “已订了酒店，帮我按酒店位置排每天路线”

## 输入要求（不全时主动追问）

- **基础信息**：目的地、出行天数、同行人数
- **时间边界**：到达/离开时间、每天可活动时段
- **预算信息**：总预算或日预算、门票/交通/餐饮偏好
- **偏好标签**：自然、人文、美食、购物、亲子、摄影、夜生活
- **强约束**：必去点、不可去点、体力限制、饮食禁忌
- **住宿位置**：酒店/民宿地址（用于路线聚类）

如果用户只给一句话，先补齐“天数 + 预算 + 偏好 + 住宿位置”四个高价值字段。

## 规划工作流

### Step 1: 旅行画像建模

提炼用户画像：
- 出行风格：打卡型 / 深度型 / 轻松型
- 节奏偏好：早出晚归 / 中等节奏 / 慢游
- 消费偏好：高性价比 / 均衡 / 品质优先

### Step 2: 候选点池与优先级

将地点分为：
- **A 类（必去）**：用户明确要求或城市核心代表
- **B 类（可选）**：同主题补充项
- **C 类（天气/状态依赖）**：可替换弹性点

### Step 3: 按日聚类与时间块排布

每一天按“区域聚类 + 时间块”设计：
- 上午：主体验（高优先级）
- 中午：就近补给（降低跨区折返）
- 下午：次体验（同片区延展）
- 晚间：轻活动（夜景/步行/自由探索）

规则：
- 单日跨区不超过 2 次（大城市可放宽到 3 次）
- 每 2-3 个活动后安排休息或机动时间
- 避免连续高体力项目

### Step 4: 预算与体力校准

为每一天给出：
- 预算区间（保守值与舒适值）
- 体力强度等级（低/中/高）
- 可删减项（时间不够时先删哪些）

### Step 5: 生成双版本结果

- **标准版**：覆盖度优先
- **轻松版**：舒适度优先（减少赶路与排队风险）

## 输出格式（强制）

1. **总览摘要**
   - 行程主题、节奏建议、总预算区间、最佳执行提醒

2. **每日计划表（Day 1...Day N）**
   - 时段、地点、活动内容、停留建议、移动方式、预计花费

3. **每日关键决策点**
   - 当天最不该错过的 1-2 项
   - 时间不够时的删减顺序

4. **备选与应急**
   - 雨天替代
   - 排队过长替代
   - 体力不足替代

5. **预订与准备清单**
   - 需要提前预订的项目
   - 出发前 24 小时检查项

## 质量门槛

输出前自检：
- 是否按天输出且每天有明确时间块
- 是否体现路线连贯性（减少折返）
- 是否给出预算区间与体力强度
- 是否提供可执行备选而非泛泛建议
- 是否对不确定信息做了假设说明

## 与多 Agent 协作建议（可选）

复杂任务推荐分工：
- **ManagerAgent**：统一约束、定义每日目标
- **POIAgent**：筛选景点与活动候选
- **ScheduleAgent**：按时间块编排日程
- **BudgetAgent**：控制预算与性价比
- **ContingencyAgent**：生成天气/延误/体力应急方案

## 闭环协同规则（Route ↔ Itinerary ↔ Budget）

当用户既关心路线又关心预算时，按以下方式闭环：

1. **接收 Route 输入**
   - 读取路线模块提供的“可达半径、跨区成本、关键换乘点”。

2. **输出给 Budget**
   - 给出每日活动清单与预算占比，标注可删减项和不可降级项。

3. **接收 Budget 回灌**
   - 根据降本要求重排日程：先替换 C 类点，再压缩 B 类点，最后才触及 A 类点。

4. **版本化输出**
   - 同时输出 `Plan-V1（原方案）` 与 `Plan-V2（预算修正）`，并列出改动影响。

## 示例触发语句

- “五一去成都 4 天，两个人，偏美食和慢节奏，帮我排每日行程。”
- “带爸妈去北京 3 天，预算适中，少走路，给我标准版和轻松版。”
- “我住在东京新宿，6 天行程，想要城市经典 + 一天近郊。”
